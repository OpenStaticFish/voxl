# VOXL ÔÇö Marketing Website

A polished, fast landing page for **VOXL**, the browser-based WebGL voxel sandbox
prototype that lives at the **root of this repository** ÔÇö and it now **serves the
playable game itself**. Clicking **Play** opens a full-screen `/play` route that
embeds the game; the site builds the game from source and ships it under `/game/`.

The game remains the single source of truth at the repo root: no engine code is
duplicated. A build step (`bun run sync:game`) compiles the game and copies its
self-contained output into `public/game/`, which Astro serves statically.

> VOXL is an original indie prototype. It is **not** Minecraft, not affiliated
> with Mojang/Microsoft, and uses no copyrighted assets. Everything you see is
> generated in code.

---

## 1. Tooling choice & justification

**Stack: Astro + TypeScript, run with Bun.**

| Choice | Why |
| --- | --- |
| **Astro** | Purpose-built for content/marketing sites. It compiles to **static HTML and ships zero client JS by default** ÔÇö exactly the performance and accessibility profile a landing page needs ÔÇö while giving a component model so a multi-section page stays maintainable (no giant `index.html`). First-class TypeScript and build-time data access lets us *honestly* detect whether real screenshots exist. |
| **TypeScript** | The content model (`src/data/site.ts`) and build-time screenshot discovery benefit from typing; strict mode catches real mistakes. |
| **Bun** | Required runtime/package manager for this repo; also runs the optional screenshot script. |

**Why not something else?**

- **Plain HTML/CSS + Vite** ÔÇö works, but a rich multi-section page becomes one
  unmaintainable file, and there's no clean component reuse. Astro gives the
  *same static output* with a component model.
- **Next.js** ÔÇö ships a React runtime the page doesn't need; SSR/routing
  machinery is overkill for a single static page. **Overkill.**
- **SvelteKit** ÔÇö excellent, but more app-oriented than a content site needs;
  adds a framework runtime where Astro adds none.

**Not overkill:** the build output is literally static HTML + CSS. **Not too
limited:** components, scoped styles, TypeScript, build-time logic, and a
~0.6 KB gzipped progressive-enhancement script for nav highlighting + a
screenshot lightbox. Dependencies: only `astro`, plus `@astrojs/check`/
`typescript` in dev.

## 2. Setup & run

Requirements: [Bun](https://bun.sh) 1.0+.

```bash
cd website
bun install          # install dependencies
bun run dev          # syncs the game, then dev server Ôćĺ http://localhost:4321
```

Open `/` for the landing page, or go straight to **`/play`** to launch the game.

> **From the repo root (runs game + site together):** `bun install` then
> `bun run dev` starts both the game's Vite HMR server (5173) and this website
> (4321) concurrently. See the root `README.md`.

### Build / preview / checks

```bash
bun run sync:game    # build the game (repo root) Ôćĺ public/game/
bun run build        # sync:game, then static build Ôćĺ website/dist/
bun run preview      # serve the build Ôćĺ http://localhost:4321
bun run typecheck    # astro check (type checks .astro + .ts)
bun run check        # alias for typecheck
```

`bun run dev` and `bun run build` both run `sync:game` first, so the embedded
game is always up to date.

### Optional visual review (Playwright)

```bash
bun add -D playwright && bunx playwright install chromium
bun run preview      # in one terminal
VOXL_SITE_URL=http://localhost:4321 bun run screenshot   # Ôćĺ website/preview.png
```

Playwright is **optional** ÔÇö the site builds and runs without it.

## 4. The playable game (`/play`)

The site doesn't just advertise VOXL ÔÇö it **runs** it. Every **Play** button
(header, hero, CTA, footer, mobile menu) opens `/play`, a dedicated full-screen
route that embeds the game in a same-origin `<iframe>`.

- **How it's served:** `bun run sync:game` builds the game at the repo root
  (the canonical source ÔÇö engine code is never duplicated) and copies its
  self-contained, relative-path build into `public/game/`. Astro serves those
  files at `/game/`, and `/play` embeds `/game/index.html`.
- **Why an iframe:** the game is a pointer-lock, full-viewport WebGL app that
  owns its entire document (canvas + DOM overlays). Embedding the built game
  same-origin keeps it the single source of truth and lets pointer lock,
  keyboard input, and focus work naturally. The iframe carries
  `allow="fullscreen; pointer-lock; ÔÇŽ"` so the permissions policy permits it.
- **Loading UX:** a veil with a spinner covers the iframe until it fires
  `load`, then focus is handed to the frame so input works immediately. A
  "Back to site" bar stays above the game.
- **The landing page stays light:** the ~132 KB-gzipped game bundle loads
  *only* when you visit `/play`. The marketing page itself ships no game code.

> Requires a desktop browser with WebGL + pointer lock (noted on the page).

## 5. Screenshots ÔÇö where they live & how they're used

The site expects two screenshots, served from the site root:

- `screenshots/main-menu.png`
- `screenshots/in-game.png`

**Place the real files at:** `website/public/screenshots/`

At **build time** (`src/data/screenshots.ts`) the site checks whether those PNGs
actually exist on disk:

- **If present** Ôćĺ a real `<img>` is rendered (with a "Real screenshot" tag and
  byte size), clickable to open a lightbox.
- **If absent** Ôćĺ a clearly-labelled **placeholder** frame is rendered instead,
  plus an amber notice telling the visitor exactly how many are missing and how
  to add them.

Nothing is faked. The repo currently ships with **no** screenshots, so the
placeholders and notice are shown. To capture real ones, see the game repo
`README.md` (press `P` in-game, or run `bun run screenshot` from the repo root),
then copy the PNGs into `website/public/screenshots/` and rebuild.

## 6. Accessibility notes

- Semantic landmarks: `<header>`, `<main>`, `<section aria-labelledby>`,
  `<nav>`, `<footer>`.
- A "Skip to content" link is the first focusable element.
- All interactive elements are real `<a>`/`<button>`; keyboard-operable with
  visible `:focus-visible` outlines.
- Mobile menu uses a native `<details>` disclosure (works without JS).
- Every image has descriptive `alt` text; decorative SVGs are `aria-hidden`.
- Colour palette is tuned for AA contrast on the dark background (ink `#e8edff`
  on `#0b1020` Ôëł 16:1; dim text kept above AA for body sizes).
- `prefers-reduced-motion` disables drifting clouds, the bobbing logo, and the
  scroll hint.
- No content is available only via hover or animation.

## 7. Performance notes

- **Zero client framework JS.** Astro emits static HTML/CSS; the only JS is a
  ~0.6 KB gzipped inline enhancement (nav active-state via
  `IntersectionObserver`, screenshot lightbox via `<dialog>`, copy buttons).
  All are progressive ÔÇö the page is fully usable without them.
- **Total landing-page weight Ôëł 21 KB gzipped** (HTML + CSS + favicon),
  measured on the production build. System font stacks only (matches the game,
  no webfont weight).
- **The game bundle loads only on `/play`.** The ~132 KB-gzipped Three.js game
  is a separate route, so the marketing page itself never pays for it; visitors
  who click Play load it on demand.
- Inline SVG for the logo, terrain dividers, favicon, and OG image (no extra
  requests).
- Images use explicit `width`/`height` + `loading="lazy"` to avoid layout
  shift.
- `compressHTML` and inlined stylesheets keep the document small.

## 10. Limitations & honest self-critique

- **No real screenshots yet.** The repo ships without PNGs, so the showcase
  honestly shows labelled placeholders + a notice. The integration is complete
  ÔÇö drop files in `public/screenshots/` and rebuild.
- **Single page.** There's no blog/docs routing; intentional for a landing page,
  but if the project grows, Astro's file-based routing makes adding pages easy.
- **System fonts over a bespoke pixel face.** Keeps it fast and offline and
  matches the game's identity, but a custom pixel font would push the "voxel"
  feel further (a trade-off against bundle weight).
- **JS lightbox is a progressive enhancement.** It adds a small amount of JS by
  choice (real UX value for a showcase); it degrades gracefully.
- The `PixelTerrain` divider is decorative (`aria-hidden`); the actual terrain
  variety is conveyed by the content and the cross-section diagram, which has a
  text alternative via `role="img"` + `aria-label`.
